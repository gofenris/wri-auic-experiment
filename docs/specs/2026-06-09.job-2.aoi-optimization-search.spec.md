## The bigger goal (why this exists)

WRI's **Cool Cities Lab** (https://coolcities.wri.org/) plans urban-cooling
interventions: cool roofs, street trees, shade parks. The normal tool works like
this: a user draws an area, and the tool reports what interventions help there.
Aman wants the reverse, which is harder:

> "the user is **not** drawing the circle. **We're** drawing the circle… I want the
> algorithm to be able to search over this data and say, here's an area where
> you'd get the most effective return on your investment if you planted trees in
> this neighborhood. That's a complicated optimization problem." :88

> "the user does not select the AOI. The algorithm selects an AOI for you and
> says, hey, go plant trees in this area." :108

This capability is durable and reusable, and it doesn't exist yet. Aman is clear
on that: he's "speccing out this new notebook… it's a search algorithm" :84, "not
building it yet" :130, and has "no time for at least a week or two." :172 It also
reaches beyond Cool Cities: "a grid search with geospatial data… would have many
different applications." :134

**One design choice decides whether this project works: how you define "best."**
Aman already flagged it: "there are some major problems with that." :128 Treat
this doc as a research bet, not a build ticket.

## This job, in one line

Score many candidate areas in a city, then return the area (or areas) where an
intervention gives the most benefit — without a human drawing the polygon.

## The task, staged

Build a naive version first, then make it real. Each stage is a milestone.

1. **Fixed-grid, additive score.** Tile the city, score every cell with Job 1's
   calculation, and pick the best. This is Aman's own baseline: "imagine if I
   divided this up into a grid and then I did a grid search… and then said, this
   is the best one." :96 Crude, but it proves the loop works end-to-end.
2. **Make the search cheap.** Once you allow *overlapping, variable-size* windows,
   the same pixels get summed over and over. The summed-area-table lever (below)
   removes that redundancy.
3. **Drop the grid for real shapes.** Segment the city into meaningful objects and
   search over those (see object-based image analysis, below). :124
4. **Make the score realistic.** Replace the toy score with one that reflects real
   ROI and constraints. This is where it gets hard.

---

## Why this is hard (the four walls)

1. **Too many degrees of freedom.** "they haven't specified how big the polygon
   should be… what shape it should be. There are too many degrees of freedom." :118
   Position × size × shape across a high-resolution city adds up to millions of
   overlapping candidate windows, each needing a raster calc. Naive search is
   intractable.

2. **The one big lever — overlap is redundant.** If the score is a **sum or mean
   over the window**, a **summed-area table (integral image)** computes any
   rectangle's score in O(1) from four corner lookups, after one precompute pass.
   Millions of evaluations collapse into one sweep. (Convolution/FFT does the same
   job if the score is a fixed kernel.) This is the technique that makes a grid
   search tractable, and it's why Job 1's statistic should be additive.

3. **Realistic scoring breaks the lever.** The integral-image trick works only
   because sums decompose. The moment the score turns nonlinear, it stops
   decomposing, the shortcut dies, and you're back to brute force or heuristics
   (greedy, submodular, simulated annealing). Nonlinear scores include diminishing
   returns on planting, a budget cap, equity weighting toward hot or poor
   neighborhoods, and "can't plant on a highway" feasibility. **These are the
   "major problems with scoring." :128** The hard part is not the search; it's the
   score function.

4. **"Any polygon" breaks the rectangle tricks too.** Aman doesn't want a grid:
   "we're doing it in technically any polygon." :96 Free-form shapes destroy the
   rectangle and kernel shortcuts entirely. And a fixed grid is "kind of dumb
   because the grid does not really correspond to anything in reality." :122

## The escape hatch Aman already named: segmentation

> "object-based image analysis [OBIA]… use the underlying morphology of this data to
> determine what the polygons should look like based on existing features. And then
> I could use those objects to do the same grid search — except now it's not a
> grid. It's searching on these segments that I've broken the city up into." :124

Segment the city into meaningful objects **first**. The search space then becomes
"which segment(s)" instead of "which of infinitely many polygons." Segmentation is
what makes the search both tractable *and* meaningful. The cost: it brings its own
parameters (scale, compactness), and the "best" segments may not match
intervention logic.

After scoring comes ranking and recommendation: "develop a scoring algorithm…
compare which one is the best… and then I can make a recommendation to the user to
say this area is the best place to plant trees." :128

## The crux (read this twice)

**Defining "best" is the project.** That one choice decides whether Job 2 is a
fast, exact computation (additive score → summed-area table) or a slow,
maybe-impossible approximation (realistic score → heuristic search). Everything
else is mechanics. Spend the design effort here, not on the search loop.

## Definition of done

Staged, because the honest outcome of stage 4 might be "this isn't cleanly
solvable":

- **M1:** fixed, non-overlapping grid, additive score, returns a defensible "best
  cell" in Cape Town, end-to-end, in a Marimo notebook. (Each pixel is read once,
  so no acceleration is needed or possible here; this is just the baseline loop.)
- **M2:** open the search to **overlapping, variable-size windows** — the realistic
  case, where the same pixels are summed many times. Show that the **summed-area
  table** makes it tractable and beats brute-force re-summing of every window.
  (The lever pays off *because* windows overlap; it does nothing for M1's disjoint
  grid.)
- **M3:** segmentation-based search over real objects, not a grid.
- **M4:** a realistic score with at least one nonlinear term or constraint;
  document where the cheap method breaks and what heuristic replaces it.

A negative result at M4 — "with a realistic score, exhaustive search is
intractable and here's the best heuristic we found" — is a legitimate, valuable
deliverable. Don't pretend otherwise.
